# Database Layer Rules

Applies to `query.sql`, `sqlc.yaml`, `migrations/**/*.sql`, and `db/**/*.go`. Persistence is SQLite via `mattn/go-sqlite3`, schema managed by Goose, queries generated by sqlc.

## Schema lives in migrations, queries in `query.sql`

- `migrations/` is the authoritative source of schema. sqlc reads it to type-check `query.sql`.
- Never duplicate schema into a separate `schema.sql`. Point `sqlc.yaml` at the migrations folder.

## Initial schema (`migrations/001_init.sql`)

```sql
-- +goose Up
CREATE TABLE todos (
  id      INTEGER PRIMARY KEY AUTOINCREMENT,
  title   TEXT    NOT NULL,
  status  TEXT    NOT NULL DEFAULT 'pending'
);

-- +goose Down
DROP TABLE todos;
```

- `status` stored as TEXT, matching `TodoState string` exactly.
- No `CHECK` constraint on status values. The FSM at the handler boundary already enforces validity, and SQLite CHECK errors are awkward to surface in HTMX responses.
- No `updated_at` / `created_at` until a feature actually needs them.

## Migration rules

- Sequential numbering: `001_init.sql`, `002_*.sql`, … Pad to three digits.
- Every migration MUST have both `-- +goose Up` and `-- +goose Down`.
- Down migrations reverse the up cleanly (tests rely on this).
- **Never edit a committed migration.** Add a new one. Even in early development, if the migration has run anywhere it ever ran, treat it as immutable.
- Migrations are embedded with `//go:embed migrations/*.sql` and run via `goose.Up(db, "migrations")` in `main.go` on every boot. That's by design — single binary, no migration step in deploy.

## Required sqlc queries (`query.sql`)

```sql
-- name: ListTodos :many
SELECT * FROM todos ORDER BY id;

-- name: GetTodo :one
SELECT * FROM todos WHERE id = ?;

-- name: CreateTodo :one
INSERT INTO todos (title, status) VALUES (?, 'pending')
RETURNING *;

-- name: UpdateTodoStatus :execrows
UPDATE todos SET status = ?
WHERE id = ? AND status = ?;

-- name: DeleteTodo :exec
DELETE FROM todos WHERE id = ?;
```

### Critical: `UpdateTodoStatus` uses optimistic locking

The `AND status = ?` clause is **not optional**. It's the database-level enforcement of the FSM:

- Caller passes `(newStatus, id, expectedCurrentStatus)`.
- If the row's current status doesn't match `expectedCurrentStatus`, zero rows are affected.
- `:execrows` returns `int64` — caller checks `rowsAffected == 0` and treats as a rejected transition.

This closes the TOCTOU window between "handler reads state" and "handler writes new state." Without it, two concurrent PUTs racing on the same todo can both see Pending and both succeed.

## `sqlc.yaml` config

```yaml
version: "2"
sql:
  - engine: "sqlite"
    schema: "migrations/"
    queries: "query.sql"
    gen:
      go:
        package: "db"
        out: "db"
        sql_package: "database/sql"
        emit_json_tags: false
        emit_prepared_queries: false
        emit_interface: false
        emit_exact_table_names: false
        emit_empty_slices: true
```

- `emit_json_tags: false` — we don't serialize to JSON anywhere.
- `emit_interface: false` — no DI interfaces; the concrete `*db.Queries` is used directly.
- `emit_empty_slices: true` — `ListTodos` returns `[]Todo{}` not `nil` when empty, simplifying templates.

After editing `query.sql` or migrations: run `sqlc generate`. Commit the regenerated `db/*.go`.

## `db/` is generated, never hand-edited

- Generated by sqlc from `query.sql` + migrations.
- Committed to source control so builds don't require sqlc to be installed.
- If a query is wrong, fix `query.sql` and regenerate. Never patch the generated code.
- If sqlc emits something you don't like (e.g., a column type), fix it via the schema or sqlc config — not by editing generated files.

## SQLite connection settings

Open with PRAGMA flags for WAL + busy timeout to handle concurrent reads gracefully:

```go
db, err := sql.Open("sqlite3", "file:todos.db?_journal=WAL&_busy_timeout=5000&_fk=on")
```

- `_journal=WAL` — readers don't block writers, important for HTMX's many small requests.
- `_busy_timeout=5000` — wait up to 5s for write locks before failing.
- `_fk=on` — foreign keys enforced. (Not needed for the initial schema, but cheap to set now.)

## What this layer does NOT do

- No ORM. sqlc + raw SQL only.
- No connection pool tuning beyond defaults. SQLite + one process needs nothing.
- No retry logic. SQLite errors propagate; let the handler turn them into a generic 500 (which for HTMX still means render an error banner).
- No multi-database support. SQLite is the database, not "a database driver."
